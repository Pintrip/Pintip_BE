# 핀트립 백엔드 개발 가이드라인 (해커톤 1일, 2인)

이 문서는 백엔드 개발자 2명이 **하루 안에 데모 가능한 MVP**를 완성하기 위한 단일 실행 가이드다.
기존 문서의 중복 내용을 통합해, 아래 순서만 보면 바로 작업을 시작할 수 있게 구성했다.

---

## 1. 목표와 범위

### 목표

- 로그인 없이 `sessionId(UUID)` 기반으로 사용자의 오늘 여행 상태를 유지한다.
- 프론트 핵심 플로우를 당일 내 연결한다.

```text
세션 생성 → 지역 선택 → 랜덤 장소 → 랜덤 퀘스트 → 기록 저장 → 새로고침 복구
```

### 당일 필수 범위 (P0)

1. `POST /trip-sessions`
2. `GET /regions`
3. `PATCH /trip-sessions/{sessionId}/region`
4. `POST /trip-sessions/{sessionId}/place/random`
5. `POST /trip-sessions/{sessionId}/quest/random`
6. `POST /trip-sessions/{sessionId}/logs`
7. `GET /trip-sessions/{sessionId}`

### 당일 제외 범위

- 인증/인가(Spring Security), 회원 기능
- Flyway, Testcontainers, Docker 배포
- 다시 뽑기 제한 고도화, 비주류 점수, 공유 기능

---

## 2. 핵심 설계 원칙

### 세션 전략 (최종)

- **UUID + DB 임시 세션** 채택
- 프론트는 `localStorage.tripSessionId` 보관
- 백엔드는 `trip_sessions`를 단일 상태 소스로 사용

### 이유

- 외부 지도 이동 후 복귀 시 쿠키 세션보다 안정적
- 다시 뽑기/완료 여부 같은 상태 검증을 서버에서 제어 가능
- 해커톤 당일 디버깅 단순화

---

## 3. 기술 스택 (최소)

- Java 17
- Spring Boot 3.x
- Spring Web, Spring Data JPA, Validation
- H2 in-memory (`schema.sql`, `data.sql`)
- (선택) Swagger springdoc-openapi

---

## 4. 백엔드 폴더 구조

패키지 트리, API↔패키지 매핑, 충돌 방지 룰은 **`백엔드_폴더_구조.md`** 를 본다.

### `application.yaml` 최소 예시 (H2)

```yaml
spring:
  application:
    name: pintrip
  datasource:
    url: jdbc:h2:mem:pintrip;MODE=MySQL;DB_CLOSE_DELAY=-1
    driver-class-name: org.h2.Driver
    username: sa
    password:
  jpa:
    hibernate:
      ddl-auto: none
    open-in-view: false
  sql:
    init:
      mode: always
      schema-locations: classpath:schema.sql
      data-locations: classpath:data.sql
```

> MySQL(`compose.yaml`)은 시간 남을 때만 전환한다. 해커톤 당일은 H2 고정을 권장한다.

## 5. 도메인 모델 요약

상세 ERD는 `핀트립_ERD.md`를 기준으로 한다.

- `trip_sessions` (PK: `id` UUID)
  - `region_id`, `selected_place_id`, `selected_quest_id`, `completed`, `reroll_count`, `created_at`, `updated_at`
- `trip_logs`
  - `session_id`, `content`, `created_at`
- `regions`, `places`, `quests`
  - seed 데이터 테이블

## 6. API 기능 설명 (백엔드 구현 관점)

### 1) 세션 생성

- `POST /trip-sessions`
- 기능: UUID 생성 후 `trip_sessions` row 생성
- 실패: 생성 실패 시 500

### 2) 지역 목록

- `GET /regions`
- 기능: 프론트 지역 선택 화면용 목록 제공

### 3) 지역 선택

- `PATCH /trip-sessions/{sessionId}/region`
- 기능: 세션에 `region_id` 저장
- 검증:
  - `sessionId` 존재 확인
  - `regionId` 유효성 확인

### 4) 랜덤 장소

- `POST /trip-sessions/{sessionId}/place/random`
- 기능: `region_id` 기준으로 장소 랜덤 1개 선택 후 세션 반영
- 검증:
  - 세션 존재
  - 세션에 `region_id` 존재
  - 해당 지역 장소 데이터 존재

### 5) 랜덤 퀘스트

- `POST /trip-sessions/{sessionId}/quest/random`
- 기능: 퀘스트 랜덤 1개 선택 후 세션 반영
- 검증:
  - 세션 존재
  - 퀘스트 데이터 존재

### 6) 기록 저장

- `POST /trip-sessions/{sessionId}/logs`
- 기능: `trip_logs`에 기록 추가
- 검증:
  - 세션 존재
  - `content` not blank

### 7) 세션 조회 (복구 핵심)

- `GET /trip-sessions/{sessionId}`
- 기능: region/place/quest/logs/completed 묶어서 반환
- 목적: 새로고침 및 외부 링크 복귀 시 복구

## 7. 공통 응답 규약

### 성공 응답

- API별로 필요한 데이터만 반환
- `sessionId`는 문자열(UUID)

### 실패 응답

```json
{
  "code": "SESSION_NOT_FOUND",
  "message": "sessionId를 찾을 수 없습니다."
}
```

권장 에러 코드:

- `SESSION_NOT_FOUND`
- `REGION_NOT_FOUND`
- `PLACE_NOT_FOUND`
- `QUEST_NOT_FOUND`
- `INVALID_REQUEST`

# 핀트립 백엔드 개발 가이드라인 (해커톤 1일, 2인)

이 문서는 백엔드 개발자 2명이 **하루 안에 데모 가능한 MVP**를 완성하기 위한 단일 실행 가이드다.
기존 문서의 중복 내용을 통합해, 아래 순서만 보면 바로 작업을 시작할 수 있게 구성했다.

---

## 1. 목표와 범위

### 목표

- 로그인 없이 `sessionId(UUID)` 기반으로 사용자의 오늘 여행 상태를 유지한다.
- 프론트 핵심 플로우를 당일 내 연결한다.

```text
동 목록 조회 → (프론트 랜덤) → 최종 동 전송 + 세션 생성 → 랜덤 장소 → 랜덤 퀘스트 → 새로고침 복구
```

### 당일 필수 범위 (P0)

1. `GET /dongs`
2. `POST /trip-sessions` (body: `dongId`)
3. `POST /trip-sessions/{sessionId}/place/random`
4. `POST /trip-sessions/{sessionId}/quest/random`
5. `GET /trip-sessions/{sessionId}`

### 당일 제외 범위

- 인증/인가(Spring Security), 회원 기능
- Flyway, Testcontainers, Docker 배포
- 다시 뽑기 제한 고도화, 비주류 점수, 공유 기능

---

## 2. 핵심 설계 원칙

### 세션 전략 (최종)

- **UUID + DB 임시 세션** 채택
- 프론트는 `localStorage.tripSessionId` 보관
- 백엔드는 `trip_sessions`를 단일 상태 소스로 사용

### 이유

- 외부 지도 이동 후 복귀 시 쿠키 세션보다 안정적
- 다시 뽑기/완료 여부 같은 상태 검증을 서버에서 제어 가능
- 해커톤 당일 디버깅 단순화

---

## 3. 기술 스택 (최소)

- Java 17
- Spring Boot 3.x
- Spring Web, Spring Data JPA, Validation
- H2 in-memory (`db/schema.sql`, `db/data.sql`, `application-local.yaml`)
- (선택) Swagger springdoc-openapi

---

## 4. 백엔드 폴더 구조

패키지 트리, API↔패키지 매핑, 충돌 방지 룰은 **`백엔드_폴더_구조.md`** 를 본다.

### DB 초기화 설정 (`application-local.yaml`)

로컬 프로필에서 H2 + SQL 자동 로딩이 적용된다.

- `spring.sql.init.schema-locations`: `classpath:db/schema.sql`
- `spring.sql.init.data-locations`: `classpath:db/data.sql`
- `spring.jpa.defer-datasource-initialization: true` (JPA와 SQL init 순서 보장)

> MySQL(`compose.yaml`)은 시간 남을 때만 전환한다. 해커톤 당일은 H2(`local` 프로필) 고정을 권장한다.

## 5. 도메인 모델 요약

상세 ERD는 `핀트립_ERD.md`를 기준으로 한다.

- `dongs` — 동 목록(mock 10개), 프론트 랜덤 후 최종 1개 선택
- `places` — 동별 place 3개, `image_url` 포함
- `quests` — mock 20개, `place_id` FK
- `trip_sessions` (PK: `id` UUID)
  - `dong_id`, `selected_place_id`, `selected_quest_id`, `status`, `created_at`, `updated_at`

## 6. API 기능 설명 (백엔드 구현 관점)

### 1) 동 목록

- `GET /dongs`
- 기능: 프론트 동 선택/랜덤 UI용 목록 제공 (`is_active = true`)
- 응답: `id`, `name`, `city`, `district`, `description`

### 2) 세션 생성 (최종 동 확정 시점)

- `POST /trip-sessions`
- body: `{ "dongId": 1 }`
- 기능: UUID 생성 + `trip_sessions` insert (`dong_id` 포함)
- 검증:
  - `dongId` 존재 및 활성 여부
- 실패: 잘못된 `dongId` -> `DONG_NOT_FOUND`

### 3) 랜덤 장소

- `POST /trip-sessions/{sessionId}/place/random`
- 기능: 세션의 `dong_id` 기준 place 랜덤 1개 선택 후 `selected_place_id` 저장
- 검증:
  - 세션 존재
  - 해당 동 place 데이터 존재

### 4) 랜덤 퀘스트

- `POST /trip-sessions/{sessionId}/quest/random`
- 기능: 선택된 place(`selected_place_id`)에 연결된 quest 중 랜덤 1개 선택
- 검증:
  - 세션 존재
  - `selected_place_id` 존재 (장소 먼저 뽑기)
  - 해당 place의 quest 존재

### 5) 세션 조회 (복구 핵심)

- `GET /trip-sessions/{sessionId}`
- 기능: dong/place/quest/status 묶어서 반환
- 목적: 새로고침 및 외부 링크 복귀 시 복구

## 7. 공통 응답 규약

### 성공 응답

- API별로 필요한 데이터만 반환
- `sessionId`는 문자열(UUID)

### 실패 응답

```json
{
  "code": "SESSION_NOT_FOUND",
  "message": "sessionId를 찾을 수 없습니다."
}
```

권장 에러 코드:

- `SESSION_NOT_FOUND`
- `DONG_NOT_FOUND`
- `PLACE_NOT_FOUND`
- `QUEST_NOT_FOUND`
- `INVALID_REQUEST`
